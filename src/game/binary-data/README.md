# Binary-extracted reference data

This directory contains **canonical values from the original 1993
CASTLE1.EXE / CASTLE2.EXE binaries**, recovered through reverse-engineering.
The data is provided as a reference parallel to the gameplay-tuned values
in `../monsters.ts` and `../spells.ts`.

## What's here

- `types.ts` — local types (`BinaryMonsterRecord`, `BinarySpellRecord`,
  `AttackFormula`, `AttackElement`, `BinarySchool`)
- `monsters.ts` — `MONSTERS_FROM_BINARY`: 40 monster archetypes with HP,
  AC, damage, XP, resist mask, behaviour flags
- `spells.ts` — `SPELLS_FROM_BINARY`: all 36 spells (32 player-castable +
  4 wand/staff-only) with level, mana, school, cast time, school-specific
  data and original 6-school classification
- `attack-formulas.ts` — `ATTACK_FORMULAS`: the 7 attack-spell damage
  parameters (max damage, AOE radius, element) and `RANGE_FALLOFF`,
  the bolt-spell distance miss formula
- `index.ts` — barrel exports + `findBinaryMonster()`,
  `findAttackFormula()`, `findBinarySpell()` lookup helpers

## How it integrates with the rest of the codebase

This module is purely additive — nothing in `src/game/monsters.ts`,
`src/game/spells.ts`, or `src/game/combat.ts` is changed structurally.
Where the gameplay code wants original-faithful behaviour it imports
specific helpers:

- `combat.ts` imports `RANGE_FALLOFF` to compute bolt-spell miss%
  using the original formula `(distance − 5) × 5`.
- `spell-engine.ts` (when wired up) imports `ATTACK_FORMULAS` to look
  up the exact `maxDamage` / `minDamageOrAoe` for the 7 bolt/ball spells
  by id.

The monster HP/AC values in `MONSTERS_FROM_BINARY` are NOT used to
override the playtest-tuned values in `../monsters.ts`.  They're
available for cross-reference and for any future feature that wants
to expose a "vintage stats" toggle.

## Validation against the gameplay catalog

`integration/compare.py` (in the data repository) shows that
**31 of 40 monsters share the exact XP value** with the gameplay
`MONSTERS` array — a strong cross-validation that the binary
extraction is correct.  HP almost never matches because the
existing gameplay values are intentionally rebalanced for play feel.

## Source

Generated by `integration/generate.py` in the reverse-engineering
repository; see that repo's `REPORT_PHASE3.md`, `REPORT_PHASE4.md`
and `REPORT_PHASE7.md` for the methodology and field layouts.

To regenerate, run the python script in that repo against the
`c1/data/*.json` files — it produces these `.ts` files verbatim.
